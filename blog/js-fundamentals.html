What's up with that HTML and CSS stuff?

Most websites are made of HTML and CSS files, at their most basic.
They look very similar, and work very closely with each other but the difference is basically this:

HTML is a sort of framework, a list. It's a document containing all the differnt pieces of text, images, boxes, buttons and the likes that make up the webpage. It's no more than a text document, really, a list of different elements and a rough idea of which ones are inside the others.

That's why we we create some styles with our CSS document.

On its own, a CSS file is even less exciting than an HTML document, at least we HTML we get an ugly list of parts while with CSS, you wouldn't see anything at all!

CSS files exist to tell the HTML Elements how they should look, where they should sit on the page and where they should sit compared to each other

In short, HTML describes WHAT is in your page and CSS describes HOW that stuff should be shown!

Alright then, what about that JavaScript thing?

Well, my sarcastic alter ego, that's a good question!

If you consider that HTML describes WHAT to build and CSS describes HOW it should look, JavaScript describes what to DO with your page!

HTML is what goes on your page when it loads, same with CSS. Once that little wheel stops spinning or the bar is filled, that's it. That page will never change, grow or do anything exciting.
Sure, you can make certain elements cange colour when you hover your cursor over them, mayba dropdown menu or two, but nothing genuinely dynamic - nothing that wasn't in your CSS or HTML to start with.

That is where JavaScript comes in. Instead of being a document your browser reads and sets up, it's a programming language that allows your browser to run real code, using the parts of your website in its programme.

Alrighty then, HTML describes contents, CSS tells them how to look and JS makes them do stuff. How?

Enough is never enough for you, is it?

Alright, fine. JavaScript is basically just a programming language, you set it up and runs through its Control Flow, repeats its loops, and acts just like any programme.

Control Flow? Loops?

Yes! Those!
Don't worry, they're not as scary as you think. The Control Flow is just the order that code is executed (top to bottom).

If running your code just once, from start to finish, isn't enough for you - maybe you need to count, do something multiple times or check back on previous runs - the you can use loops.

Consider making a cup of tea. You might grab a mug object, throw in a teabag, fill the mug with water and once all those are done, stir that mug 12 times and remove the teabag.

That's a few steps that have to be performed in order, plus a few repated stirs of the spoon.

But instead of making our own cup of tea (boring), let's write some snazzy JavaScript that will make it for us! It'd look something like this: 

        
var mug = {hasBag:false, hasWater:false};

makeTea() {
    mug.hasBag = true;
    mug.hasWater = true;
    for(i=0; i < 12; i++) {
        stirSpoon();
    };
    mug.hasBag = false;
};

Okay, let's walk through what all that means.

That first line creates a new mug with a couple properties. It's a smart mug, so it knows if it has a tea bag and some water in it, but it starts empty.

That next chunk is a set of instructions, called a function. Functions are handy because they're a set of instructions that all happen together, which we can trigger in lots of ways. We could just request some tea by typing in a command, or it could happen at a set time, or when a microphone hears us ask. The sky is the limit!
Inside our function is where the instructions are, so what do they do?

Well first, we tell the mug that it now has a teabag, then we tell it it has some water, remember the mug knows if it does or doesn't have those things, we're just changing that.

Now is where we play around, up until now we were letting the flow do its thing, one instruction after another down the page.
Now, we're creating a loop, we're creating a repeating piece of code that does whatever it contains until it's counted to the limit (in this case, 12).

That means our makeTea function will call a stirSpoon function 12 times, so we know it's properly stirred. After it's finished with the loop, our code goes back to its normal flow and works down the list.
It goes to the next command, where it removes the teabag. We've left this outside our loop because we don't want it trying to remove the bag 12 times, once every stir, because that'd be weird
The final curly brackets at the bottomm end our function and lets our code finish up.

So what did we do? We made some code, which will work through step by step in its normal flow, doing each task once, until we need it to repeat something a few times so we break that flow with a loop!

Got it! But what was that about Functions?

A function, dear friend, is basically a command that you can call on in other parts of your code.

That code up top was for a makeTea() function, which means we can have other pieces of code that can ask makeTea() to run, then carry on with what it was doing!

So what goes into a function?

Well, first we have the name, in this case that's "makeTea". Then we have a pair of brackets.
Inside those brackets we can make our function expect certain pieces of information, called arguments. With arguments, we can pass information over to our function that it can use when it does its thing.

Let's do an example. Imagine we want to make that cup of tea, but we want to be able to tell our function how many times it should stir the spoon.

We'll have to rewrite our function so that it can receive an argument and use that to decide how much to stir.

That would look like this: 

        
makeTea(stirs) {
    mug.hasBag = true;
    mug.hasWater = true;
    for(i=0; i < stirs; i++) {
        stirSpoon();
    };
    mug.hasBag = false;
};

See the difference? We've put "stirs" as an argument of MakeTea(). So instead of just calling makeTea() and it giving us the same cup every time, we now call the function and give it a number as its argument. So now we can call makeTea(12) and our tea maker will read our number of stirs, and use that as the number it'll count to! And yes, we did just ask it to stir the same number of times as it already did...

This might seem like a minor change, but the less we explicitly code in numbers, amounts, words and so on, the better. This means our code can change, work as needed and if we found we like our tea with more stirs, we don't have to go through and change the number every time.

Okay, so a function is just a task we can request using its name, without writing the instructions every time?

Exactly! Repeating yourself in your code is messy and confusing, better to have lots of small functions than a few big ones

But I have friends, what if I want to make more than one cup of tea and make them all different?

...Okay. I feel like you're asking these questions just to help a blog post flow.

But fine! Let's say we have five friends, each of whom like their tea stirred a different number of times.

We could just call our makeTea() function five times and give it a different stir argument every time, that would look like this: 

        
makeTea(5);
makeTea(7);
makeTea(9);
makeTea(3);
makeTea(15);

There you go, we've asked our tea maker to do its thing 5 times over and given it different numbers each time

But that's pretty messy, and we're repeating ourselves so we should probably ask our tea maker to read the orders from a list.
There are a few ways we can try, but how about we use an array.

An array as in a list?

Kind of. Think of them more as a spreadsheet that we can hold information in.

Let's make an array of all those different tea orders, it would look like this: 

        
var stirOrders = [5, 7, 9, 3, 15];

There we are! We've declared stirOrders and by using those square brackets, we've made it into an array. For simplicity, we've thrown our list of orders straight into the array.

Now we have to read our order list. Again, there are a few ways to do it but we'll make a quick loop that'll make a cup of tea for every order in the list

Here goes: 

        
for(i=0; i < stirOrders.length; i++) {
    makeTea(stirOrders[i]);
};

Simple as that!

Our new loop uses the length of our order list as its maximum, so it will make orders one by one until it's worked through our array.

It's important to remember that JavaScript arrays are 0 indexed, the first entry is number 0 and so on, which is why we start our count at 0 and end it 1 before the full length.

Now that we're counting an array, we need to access the information inside it.
Since our counting variable "I" is the same number as our position in our array, we can use that to tell our tea maker which stir request it'll use.

To read data from an array we use arrayName[theEntryWeWant], and we've put that code as the argument of each makeTea call!

Okay, an array is a list and we can access information inside it. Didn't you mention objects?

I did! How about instead of a list of orders, we make each person an object and give them a stir preference?

First, we'll make a person who only has two traits, their name and how they like their tea: 

        
var person = {name:"", stir: 0};

There we go, we've made a "person" object and decided they'll have a string for their name and a number for their preferred stirs.

Like before, let's make an array containing our five friends: 

          
let people = [
    {
        "name": "Geoff",
        "stirs": 5,
    },
    {
        "name": "Lauren",
        "stirs": 7,
    },
    {
        "name": "Beth",
        "stirs": 9,
    },
    {
        "name": "Bob",
        "stirs": 3,
    },
    {
        "name": "Phoebe",
        "stirs": 15,
    },
];

There you go, we've made an array of people containing our five friends.

Now we can loop through each person, just like the previous array, and access their stirs properties. 

          
for(i=0; i < people.length; i++) {
    makeTea(people[i].stirs);
};

Just like last time, we're cycling through our array of 5 items, but this time those items are people instead of numbers.
For each person, we access their stirs property by using either object.property or object[property] (we've used dots so it's easier to read).

Seems like a lot of work, what's the difference between using data from arrays and objects?

An array would work just fine if we only wanted a list of tea orders, but we're only making a list of orders, we're not making tea for a list of people

Objects, on the other hand, can have loads of properties! What if each person could decide the temperatur, type of tea, decide if they want a keep cup and decide their cup colour? Our array suddenly gets very big and very complicated, and we can still only ask for a person's order by knowing their place in our list

An object means we can just make tea for a certain person by using their name, make it just how they like it and do it whenever we like!

Right... Flows, loops, functions, arrays and objects. How is this relevant to a website?

And that's the important part!

Our JavaScript can interact directly with out HTML and CSS code by looking at the DOM.

The Dom? Like a mafia boss?

No, not a Don, the DOM. The DOM (or Document Object Model) is a representation of all the objects that make up a webpage.
All those Divs, Imgs, paragraphs and headings? They're all viewed as objects by the DOM, and the DOM is how our code sees our page.
Because of our DOM, our JavaScript can access elements on our page without having to read the HTML code and figure it all out.

Let's do an example, let's say we want to run a JavaScript function when our page loads. In fact, let's create a link that'll ask for a cup of tea when we click it! 

          
window.onload = makeTea() {
    mug.hasBag = true;
    mug.hasWater = true;
    for(i=0; i < 12; i++) {
        stirSpoon();
    };
    mug.hasBag = false;
};

There! We've made some JavaScript that'll access our DOM and once the whole DOM has loaded, it'll run our makeTea() function and brew us a lovely cup.

Our scripts can also access individual page elements, seeing each one as an object, make changes to them, make our content dynamic and so much more.

In short, the DOM is the model we use to describe our HTML and CSS in a way that can be viewed, accessed and changed by our code.
Anything we can do in JavaScript, we can do to our websites by access the DOM!
