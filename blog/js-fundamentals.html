What's up with that HTML and CSS stuff?

Most websites are made of HTML and CSS files, at their most basic.
They look very similar, and work very closely with each other but the difference is basically this:

HTML is a sort of framework, a list. It's a document containing all the differnt pieces of text, images, boxes, buttons and the likes that make up the webpage. It's no more than a text document, really, a list of different elements and a rough idea of which ones are inside the others.

That's why we we create some styles with our CSS document.

On its own, a CSS file is even less exciting than an HTML document, at least we HTML we get an ugly list of parts while with CSS, you wouldn't see anything at all!

CSS files exist to tell the HTML Elements how they should look, where they should sit on the page and where they should sit compared to each other

In short, HTML describes WHAT is in your page and CSS describes HOW that stuff should be shown!

Alright then, what about that JavaScript thing?

Well, my sarcastic alter ego, that's a good question!

If you consider that HTML describes WHAT to build and CSS describes HOW it should look, JavaScript describes what to DO with your page!

HTML is what goes on your page when it loads, same with CSS. Once that little wheel stops spinning or the bar is filled, that's it. That page will never change, grow or do anything exciting.
Sure, you can make certain elements cange colour when you hover your cursor over them, mayba dropdown menu or two, but nothing genuinely dynamic - nothing that wasn't in your CSS or HTML to start with.

That is where JAvaScript comes in. Instead of being a document your browser reads and sets up, it's a programming language that allows your browser to run real code, using the parts of your website in its programme.

Alrighty then, HTML describes contents, CSS tells them how to look and JS makes them do stuff. How?

Alright, fine. JavaScript is basically just a programming language, you set it up and runs through its Control Flow, repeats its loops, and acts just like any programme.

Control Flow? Loops?

Yes! Those!
Don't worry, they're not as scary as you think. The Control Flow is just the order that code is executed (top to bottom).

If running your code just once, from start to finish, isn't enough for you - maybe you need to count, do something multiple times or check back on previous runs - the you can use loops.

Consider making a cup of tea. You might grab a mug object, throw in a teabag, fill the mug with water and once all those are done, stir that mug 12 times and remove the teabag.

That's a few steps that have to be performed in order, plus a few repated stirs of the spoon.

But instead of making our own cup of tea (boring), let's write some snazzy JavaScript that will make it for us! It'd look something like this:

                
var mug = {hasBag:false, hasWater:false};

makeTea() {
    mug.hasBag = true;
    mug.hasWater = true;
    for(i=0; i < 12; i++) {
        stirSpoon();
    };
    mug.hasBag = false;
};

Okay, let's walk through what all that means.

That first line creates a new mug with a couple properties. It's a smart mug, so it knows if it has a tea bag and some water in it, but it starts empty.

That next chunk is a set of instructions, called a function. Functions are handy because they're a set of instructions that all happen together, which we can trigger in lots of ways. We could just request some tea by typing in a command, or it could happen at a set time, or when a microphone hears us ask. The sky is the limit!
Inside our function is where the instructions are, so what do they do?

Well first, we tell the mug that it now has a teabag, then we tell it it has some water, remember the mug knows if it does or doesn't have those things, we're just changing that.

Now is where we play around, up until now we were letting the flow do its thing, one instruction after another down the page.
Now, we're creating a loop, we're creating a repeating piece of code that does whatever it contains until it's counted to the limit (in this case, 12).

That means our makeTea function will call a stirSpoon function 12 times, so we know it's properly stirred. After it's finished with the loop, our code goes back to its normal flow and works down the list.
It goes to the next command, where it removes the teabag. We've left this outside our loop because we don't want it trying to remove the bag 12 times, once every stir, because that'd be weird
The final curly brackets at the bottomm end our function and lets our code finish up.

So what did we do? We made some code, which will work through step by step in its normal flow, doing each task once, until we need it to repeat something a few times so we break that flow with a loop!
