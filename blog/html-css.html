Classes vs IDs


We're writing some HTML code, making a sweet webpage like the one you're on we're having a good time, got a few snazzy DIVs, a couple IMGs and maybe a few paragraphs for flavour.

But wait! All our little pieces (or elements if you're feeling fancy) are sitting on the screen in a big vertical line, they're all plain and boring and while they have all the information, they don't look very good!
We're going to need to jazz them up a little, we're going to give them some style. 


          h1 {
            font-family: "Zen Dots";
            font-size: 30px;
          }
          h2 {
              font-family: "Roboto";
              font-weight: bold;
              font-size: 20px;
          }
          p {
              font-family: "Roboto";
              font-size: 18px;
          }

Yeah, like that!
In fact, those are some of the styles for the page you're seeing right now! 😉

If we take a look at that formatting, we can see some pieces that make sense; font-size does what it says, we can see names of fonts next to "font-family" and so on.

But what's with those little h1 and h2 bits? They're not in the brackets, so they must be important, right?
Well they are! Those little headings are how we reference specific elements in the HTML code (the one that builds stuff) from our CSS code (the one that tells it how to look and behave).

But what does h1 mean?
Well, h1 is a built in part of HTML, it's the highest level (meaning biggest, most important) heading - just like the ones you can choose in Word.
There's an h1 heading at the top of this page, you're reading a p (or paragraph) right now.

Getting the gist? CSS tells HTML elements how to look and behave, and elements need labels so the HTML knows where to look in the CSS.

But what if we want to apply some CSS to something that ISN'T built into HTML? What if we want to apply a style to that snazzy block of code you just scrolled past?

Well for that, we need to describe that content, and for that we use either an ID or a Class, collectively these are known as attributes.

How does an attribute work?

Just like you'd expect! When we create an element, we give it attributes to label it as something, so that when our HTML code looks for some styles it will know how to find them.
For example, this jumble of words lives inside a div, essentially a container to hold content.

But a Div is just a block, it doesn't look like anything on its own. So let's make our Div all nice and pretty. For that, we need to give it some style in our CSS file.
But before that can happen, we need to give it a name so it knows where its styles are.

Yes, we really did just switch a "." for a "#", that's how easy it is!

So what's the difference?

Since an ID and a Class serve basically the same function, can be swapped by changing a whole one character on the HTML and the CSS files, what's different about them? Why should you use one and not the other?

Well...

An ID is unique and a Class is not

There's your answer. An element can only have one ID, and an ID can only apply to one element on any given page, while an element can have many classes and those classes can apply to many elements over and over.

This makes an ID very useful for something that appears once on a page (like the post-container that holds everything you see) while a class is useful for applying code to an element that pops up lots of times (like the code-container holding those lovely code snippets).

Technically, CSS doesn't actually care which one you use, it can give the same styling to an ID or a Class. They achieve basically the same function when it comes to applying styles.

So it's more of a guideline than a rule?

Nope! CSS may not care which you use, but some things will...

Your browser

Your browser does notice the difference. The key is that an ID is much more specific, your browser knows that a Class can show up all over, so it basically ignores them, but it knows that IDs matter and will look for them on your page.
This lets you do things like link directly to an ID on a page. That TL;DR link up at the top does just that, it links to that big bold section down below just by looking for the "tldr" ID. Nifty, huh>

Scripts as well

Want to use JavaScript? Of course you do! It does fancy, fun things that you wouldn't do with HTML

But have you been throwing the same ID all over your page?
Well you're not going to have a good time... JavaScript can be told to look for an element by ID, much like your browser, but if that ID shows up in a bunch of elements, it'll have no idea which one you really wanted it to play with and your website will be even more of a mess than this one...

IDs and Classes do the same job, they're written into HTML to tell other code where things are and where to do their work.
Their difference is in how they're used, IDs are unique and for one element each while Classes aren't an can be used over and over for repeated elements.
