First up, a super simple problem that had me tearing my hair out:

Picture this, I’ve just spent half an hour writing code, I’m trying to make a JavaScript function that’ll run a dropdown menu. After getting it all written and what appears to be correct, I just cannot get the thing to work. It’s late, I’m tired and this is getting painful.

Want to know what the problem was? 

I’d spelled a single word wrong! Instead of ‘drop-content’ I’d written ‘dorp-content’, and I couldn’t spot the typo for the life of me. I definitely felt like a dorp an hour later after I’d gone away, chilled out for a bit, came back fresh and finally noticed my problem.

It wasn’t a big problem, it wasn’t the end of the world, but I was looking for some clever trick in my code to convince it to work and I neglected to look at the most simple solution!

Let’s look at a more elegant solution

Invisible typos aside, I ran into an interesting solution while working through the JavaScript Kata, specifically the Gradebook exercise.

The instructions asked for each student to be declared an object and for each to have their scores as a property. Writing them out one by one in a long list felt like a lot of code to accomplish something simple, so I wrote a simple loop to create them all at once!

A more elegant solution, sure, but it did cause some confusion when I reached the stretch task that asked me to do exactly that! After a few minutes of being puzzled at why it was asking for the same thing twice, I realised I could simply copy and paste my loop into this new function and get it done.

Not only had I solved the first task more simply than intended, but I’d solved the stretch task at the same time! Feels pretty elegant to me…

So what techniques can I use to solve problems?

Loads! Want a list? I’m giving you a list:

  • Pseudocode - The simplest, and my favourite! It’s a fancy sounding name, but all you’re really doing is writing out your intentions in comments, rather than diving into the code straight away. So you might write a list of comments describing what you know you need each function to do, what it requires and what it should return, then going through and figuring out how it will do it with your real code.As functions get longer and more complicated, holding their whole structure in your head gets hard, so writing it out first and filling in the blanks is the way to go.
  • Give it a try - Does what it says on the tin! A little like the opposite to pseudocode, if you have a rough idea of what you need to do and how it could work, just start bashing the code in there! Keep running it and checking the console for errors and trial and error your way to the finish line.Don’t worry about keeping your code neat and tidy, just focus on making it work. Once it does what it needs to do, you can go back, refactor and get it looking how it should.
  • The rubber ducky! - Fun name, but a seriously clever practice. Imagine there’s a rubber ducky on your desk (or get a real one!). He knows nothing about coding, and you’ve got to explain exactly what your code does, line by line, in as much detail as possible.Sometimes, when your code isn’t working how you want, forcing yourself to slow down and really understand every line you’ve written is what helps you spot those bugs
  • Read your error messages! - Another one I rely on heavily. You’ve written some code, you’ve run it and it got halfway and ended up stuck, or it displayed something super weird you didn’t intend, or nothing showed up. Sure you could go through line by line and explain it all to our lovely duck, OR you could check your console and see what it’s telling you. That error at line 38 saying it didn’t expect a right bracket? It’s probably telling you that right bracket isn’t meant to be there
  • Console.logging - Does your code rely on a return from a function, or an object working properly? If you’re returning something or using objects, it probably does!To check if they’re using the right information, sometimes it’s best to add a cheeky console.log to your code so you can display that full object or that returned information in your console (this is exactly how I found out my students in the gradebook exercise all had ‘undefined’ as their names). Try it, it works!
  • Give it a Google! - Ah, a classic. What a wonderful thing Google can be, provided you ask it the right questions.The two best sources I’ve found Google can point you to are documentation for the language you’re using (which you should probably keep open anyway) so you can remind yourself of the inputs, outputs and syntax of built in methods and coding forums. Forums are wonderful because 90% of the time, someone else has asked the same question you are and have been given the answer.Make sure you never copy and paste some code from the internet though! Read it, make sure you understand it, use what they’ve provided to inform your own code - copy and paste make make your code work but you can’t copy and paste into your brain, so you won’t learn from it.
  • Ask your peers and coaches - Everyone here is at a different level of experience, at a different stage in Foundations and being challenged by different aspects. Perhaps one of your peers is a total wizard when it comes to working with arrays, while you’re the guru on objects - why not fire them a message, join forces and help each other. Complementary skill sets are awesome, you get to combine the knowledge of multiple people, fill in each others’ knowledge gaps and help everyone around you succeed.Your coaches are even more knowledgeable, they wrote your course and come with a lot of experience, so if the sky is absolutely falling, or if you need help with an aspect of EDA rather than help with your code, they’re the ultimate source of knowledge.
  • Get reflecting! - There’s a reason we reflect on each task and each sprint - it works! Not only is it a great way to get your brain to access information and work on neuroplasticity, it forces you into a bit of meta-thinking (that’s thinking about your thinking). Look back at that last coding challenge you had, what was easy? What was a real challenge? Did you learn any neat tricks that helped you?Focus on those challenging aspects, figure out why they challenged you, learn the topics you weren’t sure on, reinforce those tricks so you will remember them next time. Every task is a chance to learn, and every chance to learn makes your next task easier
  • Make some tea! - This one wasn’t on the list, but taking a few minutes to get away from your desk, go for a walk, get a fresh perspective and open up your tunnel vision is an awesome way to get past a roadblock.Plus I wanted another tea reference in here.
